Ingestion
alwaysStore raw data exactly as received — no value transformations
Preserve source format: JSON blobs, CSV rows, API payloads as-is
alwaysAdd ingestion metadata columns
_ingested_at (timestamp), _source_file or _source_system, _batch_id
alwaysChoose append-only or upsert strategy — document it
Append-only = full history; upsert = idempotent re-runs. Pick one.
commonStore as Parquet or Delta/Iceberg with schema enforcement off
Allows schema drift from source without breaking the pipeline
Observability
alwaysLog row counts per batch and alert on zero-row loads
Silent empty loads are the hardest bugs to catch downstream
commonRecord schema of each batch (column names + data types)
Catches source-side schema changes before they break Silver
commonCapture file/API checksum or hash for dedup detection
Avoid re-processing identical files sent twice by the source
Access & retention
alwaysRestrict write access — only the ingestion pipeline writes here
Prevents accidental manual edits that break reproducibility
commonSet a retention/archival policy (e.g. 90 days hot, then cold storage)
Bronze grows fast — plan for storage costs from day one
optionalPartition by date or source for faster downstream reads
e.g. /bronze/orders/year=2024/month=03/

Deduplication
alwaysDefine the natural key for each entity and deduplicate on it
e.g. order_id, user_id + email. Keep latest or highest-priority row.
commonHandle slowly changing dimensions (SCD) — decide on Type 1, 2, or 3
Type 2 = keep history with valid_from/valid_to. Type 1 = overwrite.
Data type casting
alwaysCast all date/datetime columns to a single standard format (ISO 8601)
Consolidate DD/MM/YYYY, MM-DD-YY, Unix timestamps → YYYY-MM-DD
alwaysCast numeric strings to correct numeric types
"59.99" → DECIMAL(10,2). Beware locale issues: "1.234,56" vs "1,234.56"
commonStandardise timezone — store all timestamps in UTC
Add a separate local_tz column if the source timezone is meaningful
String normalisation
alwaysNormalise enum / categorical values to a consistent case
"PENDING", "Pending", "pending" → single canonical value
commonTrim leading/trailing whitespace on all string columns
Common in CSV exports — "Alice " ≠ "Alice" in joins
commonMap free-text to a controlled vocabulary where needed
e.g. country name → ISO 3166 code, currency name → ISO 4217
Null & validity handling
alwaysReplace placeholder nulls with proper NULLs
"N/A", "NULL", "-1", "" → NULL. Document each substitution.
alwaysApply range/domain validation and quarantine invalid rows
Negative amounts, future birthdates, unknown enum values → quarantine table
commonCheck referential integrity (FK lookups) and flag orphan rows
Order rows pointing to non-existent customer_id → log + quarantine
Lineage & metadata
alwaysAdd _updated_at and _source_bronze_id for row-level lineage
Enables tracing any Silver row back to its Bronze origin
commonWrite rejected rows to a separate bad-data table with reason codes
Never silently drop bad rows — make them visible and fixable

Joins & enrichment
alwaysJoin fact tables to dimension tables — no raw IDs in Gold
customer_id → customer_name, country. product_id → product_name, category.
commonApply currency / unit conversion to a single standard
All amounts in one currency, all weights in kg, all distances in km
optionalEnrich with third-party data (geo IP lookup, firmographic data, etc.)
Avoid in Silver — enrichment APIs can be slow or costly
Aggregation & derived metrics
alwaysPre-aggregate to the grain your consumers actually query
Daily sales per store, monthly revenue per customer, etc. Match BI needs.
alwaysCompute and name business metrics explicitly
GMV, LTV, churn rate, conversion rate — with agreed definitions in a data dictionary
commonAdd time-intelligence columns: day_of_week, is_weekend, fiscal_quarter
Prevents analysts from re-deriving these inconsistently across reports
Data quality for consumers
alwaysApply row-level security or column masking for sensitive fields
PII (email, phone, SSN) should be masked or excluded by default
commonAdd data freshness metadata (_as_of_date, _pipeline_run_id)
Lets dashboards show "data as of 08:00 UTC" instead of silently stale
commonRun automated DQ tests before publishing (row count, nulls, ranges)
Block the pipeline if KPIs drop more than X% vs prior run
Performance & access
commonPartition and/or cluster on the most common filter columns
Partition by date if time-series; cluster by customer_id or region
optionalMaterialise as view vs table based on query frequency
High-frequency queries → materialised table. Low-frequency → view is fine.
